## 20:00+ UTC - Orchestrator 57-Minute Timeout Investigation

### Problem
- his-orchestrator cron run at 18:17 UTC took 57 minutes (3438s) to timeout
- Session: kind-breeze, jobId: e822bc19
- Timeout at 19:26 UTC

### Root Cause Found (from subagent logs)

OpenClaw subagent documentation states:
> "After spawning children, do NOT call sessions_list, sessions_history, exec sleep, or any polling tool. Wait for completion events to arrive as user messages"

The Orchestrator violated this pattern:
1. `sessions_spawn` is **async** - returns immediately with `{ status: "accepted", runId, childSessionKey }`
2. After spawning, Orchestrator called `process poll(timeout: 60000)` to wait
3. The spawned commands may not complete within the poll timeout
4. Cron job ran from 18:17 to 19:26 = 69 minutes total

### Sessions_spawn Behavior (confirmed by docs)
- Non-blocking: returns `{ status: "accepted", runId, childSessionKey }` immediately
- Sub-agent runs in its own session `agent:<id>:subagent:<uuid>`
- On completion: announces result back to requester chat channel
- No need for parent to wait/poll

### Correct Orchestrator Pattern
```
1. sessions_spawn(sub-agent)
      ↓
2. Immediately return (do NOT wait)
      ↓
3. Wait for completion event to arrive as user message
      ↓
4.收到 event 后处理下一个
```

### NOT recommended:
- ❌ `process poll` after spawn
- ❌ `sessions_list` after spawn  
- ❌ `exec sleep` after spawn
- ❌ `sessions_history` after spawn
